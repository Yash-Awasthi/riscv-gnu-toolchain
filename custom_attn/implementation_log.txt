  RISC-V Custom `attn` Instruction — Implementation Log
═══════════════════════════════════════════════════════════════

Project: Custom RISC-V instruction for Transformer attention
Instruction: attn rd, rs1, rs2
Operation: Attention(Q, K, V) = softmax(Q × K^T / √d_k) × V
Encoding: R-type, opcode 0x0b (custom-0), funct7=0x01
  MATCH_ATTN = 0x0200000b
  MASK_ATTN  = 0xfe00707f

Toolchain: riscv-gnu-toolchain (GCC 15.2.0, binutils)
Target: riscv64-unknown-elf
Build: ~/riscv/bin/riscv64-unknown-elf-gcc

═══════════════════════════════════════════════════════════════
  GIMPLE PASS — Technical Details & Fixes Applied
═══════════════════════════════════════════════════════════════

The pass (riscv_attn_detect) runs after loop optimization and
performs the following:

1. DETECTION PHASE
   — Collects top-level loops from the function's loop tree
   — Reverses loop order (GCC lists children in reverse program order)
   — Slides a window of 4 consecutive loops, checking:
     Stage 1: is_matmul_pattern()     — triple-nested, MULT+PLUS reduction
     Stage 2: is_elementwise_div()    — double-nested, MULT by 1/sqrt(d)
     Stage 3: is_softmax_pattern()    — outer loop with 2-3 child loops
                                        (exp+sum and normalize)
     Stage 4: is_matmul_pattern()     — triple-nested, same as Stage 1

2. REPLACEMENT PHASE
   — Extracts function parameters (n, d, Q, K, V)
   — Builds stack structs: dims{n,n,n,d} and qkv{Q,K,V}
   — Emits __builtin_riscv_attn(&dims, &qkv) call
   — Redirects control flow to skip original loops
   — Dead code elimination (DCE) cleans up unreachable loop bodies

Fixes applied during integration with GCC 15.2.0:

  FIX 1: TARGET_MEM_REF (tree code 166) vs MEM_REF (167)
    GCC 15 uses TARGET_MEM_REF for pointer-based memory accesses.
    Added TARGET_MEM_REF checks alongside MEM_REF in:
    — find_matmul_reduction() for array operand extraction
    — is_elementwise_div() for scale array detection
    — strip_to_decl() for base pointer tracing
    — is_matmul_pattern() store target scanning

  FIX 2: PLUS_EXPR operand ordering
    GCC may place the MULT_EXPR result in either rhs1 or rhs2
    of the accumulation PLUS_EXPR. Added checks for both positions.

  FIX 3: Loop ordering (reverse program order)
    GCC's loop tree lists child loops in reverse program order.
    Added reversal of top_loops vector before pattern matching.
    Guard: skip reversal when top_loops.length() <= 1 to prevent
    unsigned underflow.

  FIX 4: Softmax 2-child support
    Simple softmax (without max-reduction for numerical stability)
    produces only 2 child loops (exp+sum and normalize), not 3.
    Added auto-detection of which child has exp() calls to handle
    both orderings.

  FIX 5: Store target extraction
    The matmul accumulator (sum SSA) is in the innermost loop, but
    the actual store to the score array is in the middle loop.
    Added scanning of the middle loop for memory stores.

  FIX 6: Function parameter extraction for replacement
    SSA base pointers from pattern matching are deep in address
    arithmetic chains. Replacement uses DECL_ARGUMENTS to get
    the original function parameters (n, d, Q, K, V) directly.

  FIX 7: Header dependencies
    — Added #include "cfghooks.h" for redirect_edge_and_branch()
    — Removed #include "tree-data-ref.h" (unused, caused ordering issues)
    — Moved #include "fold-const.h" before tree-scalar-evolution.h
    — Added #include "context.h" to riscv.cc for gcc::context

═══════════════════════════════════════════════════════════════
  SUMMARY
═══════════════════════════════════════════════════════════════

Two paths to the attn instruction:

  Path A: __builtin_riscv_attn(dims, qkv)
    → Direct, explicit. Works in all cases.

  Path B: Write plain C loops matching the attention pattern
    → Compiler auto-detects and replaces with attn instruction.
    → Requires: 4 separate stages (matmul, scale, softmax, matmul)
    → -O2 optimization level required for the pass to run.

Both paths produce identical machine code: attn rd, rs1, rs2
(encoding 0x02b5050b).
