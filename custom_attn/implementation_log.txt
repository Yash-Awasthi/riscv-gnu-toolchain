  RISC-V Custom `attn` Instruction — Implementation Log
═══════════════════════════════════════════════════════════════

Project: Custom RISC-V instruction for Transformer attention
Instruction: attn rd, rs1, rs2
Operation: Attention(Q, K, V) = softmax(Q × K^T / √d_k) × V
Encoding: R-type, opcode 0x0b (custom-0), funct7=0x01
  MATCH_ATTN = 0x0200000b
  MASK_ATTN  = 0xfe00707f

Toolchain: riscv-gnu-toolchain (GCC 15.2.0, binutils)
Target: riscv64-unknown-elf
Build: ~/riscv/bin/riscv64-unknown-elf-gcc

═══════════════════════════════════════════════════════════════
  GIMPLE PASS — Detection Phase
═══════════════════════════════════════════════════════════════

The pass (riscv_attn_detect) runs after GCC's loop optimization
phase and performs the following:

1. Collects top-level loops from the function's loop tree
2. Reverses loop order (GCC lists children in reverse program order)
3. Slides a window of 4 consecutive loops, checking:
  Note: At -O2, GCC fuses the scale operation into the matmul loop. In practice
  the pass detects 3 top-level loops (matmul+scale fused, softmax, matmul2).
  The Stage 2 detector (is_elementwise_div) handles the unfused case.


  Stage 1: is_matmul_pattern()
    — Triple-nested loop (i, j, k)
    — Inner loop has MULT_EXPR + PLUS_EXPR accumulation pattern
    — Detects: score[i][j] += Q[i][k] * K[j][k]

  Stage 2: is_elementwise_div()
    — Double-nested loop (i, j)
    — Body contains MULT_EXPR by a scalar (1/sqrt(d))
    — Detects: score[i][j] *= scale

  Stage 3: is_softmax_pattern()
    — Outer loop (i) with 2–3 child loops
    — Child loops contain __builtin_expf() calls + RDIV_EXPR
    — Detects: exp(score), sum, normalize per row

  Stage 4: is_matmul_pattern()
    — Same as Stage 1
    — Detects: output[i][j] += score[i][k] * V[k][j]


═══════════════════════════════════════════════════════════════
  GIMPLE PASS — Replacement Phase
═══════════════════════════════════════════════════════════════

When all 4 stages are detected in sequence:
  The replacement function (riscv-attn-detect.cc) implements the full
  struct-passing calling convention described below.


1. Extract function parameters via DECL_ARGUMENTS:
   — n (int), d (int), Q (float*), K (float*), V (float*)

2. Build dims struct on stack:
   — {rows=n, cols=n, seq_len=n, d_model=d}

3. Build qkv struct on stack:
   — {Q, K, V} (float pointers)

4. Split preheader→loop1 edge via split_edge():
   — Creates a fresh basic block for the instruction

5. Emit attn instruction via volatile inline assembly:
   — .insn r 0x0b, 0, 0x01, x0, %0, %1
   — Uses .insn directive (NOT __builtin_riscv_attn)
   — Marked volatile to prevent DCE removal

6. Find exit block via manual edge scanning:
   — single_exit(l4) may return NULL in GCC 15
   — Fallback: scan loop 4's blocks for edges leaving the loop
   — Last resort: use the function's EXIT_BLOCK predecessor

7. Redirect control flow:
   — New block's successor redirected from loop1 header to exit block
   — All 4 original loops become unreachable dead code
   — GCC's built-in DCE cleans up the dead loop bodies


═══════════════════════════════════════════════════════════════
  FIXES APPLIED — GCC 15.2.0 Compatibility
═══════════════════════════════════════════════════════════════

FIX 1: TARGET_MEM_REF (tree code 166) vs MEM_REF (167)
  GCC 15 uses TARGET_MEM_REF for pointer-based memory accesses
  after loop optimization. Added TARGET_MEM_REF checks alongside
  MEM_REF in all pattern-matching functions:
  — find_matmul_reduction() for array operand extraction
  — is_elementwise_div() for scale array detection
  — strip_to_decl() for base pointer tracing
  — is_matmul_pattern() store target scanning

FIX 2: PLUS_EXPR operand ordering
  GCC may place the MULT_EXPR result in either rhs1 or rhs2
  of the accumulation PLUS_EXPR. Added checks for both positions.

FIX 3: Loop ordering (reverse program order)
  GCC's loop tree lists child loops in reverse program order.
  Added reversal of top_loops vector before pattern matching.
  Guard: skip reversal when top_loops.length() <= 1 to prevent
  unsigned underflow (crashed the compiler on ALL functions
  including newlib, breaking the entire toolchain build).

FIX 4: Softmax 2-child support
  Simple softmax (without max-reduction for numerical stability)
  produces only 2 child loops (exp+sum and normalize), not 3.
  Added auto-detection of which child has exp() calls to handle
  both orderings.

FIX 5: Store target extraction
  The matmul accumulator (sum SSA) is in the innermost loop, but
  the actual store to the score array is in the middle loop.
  Added scanning of the middle loop via loop_outer(innermost)
  for memory stores.

FIX 6: Function parameter extraction for replacement
  SSA base pointers from pattern matching are deep in address
  arithmetic chains (e.g., ssa_name(-1)). fold_convert() on these
  caused segfaults. Replacement uses DECL_ARGUMENTS to get the
  original function parameters (n, d, Q, K, V) directly.

FIX 7: Header dependencies
  — Added #include "cfghooks.h" for redirect_edge_and_branch()
  — Removed #include "tree-data-ref.h" (unused, caused ordering issues)
  — Moved #include "fold-const.h" before tree-scalar-evolution.h
  — Added #include "context.h" to riscv.cc for gcc::context *g

FIX 8: Builtin lookup — NULL gaps in target builtin table
  The RISC-V target builtin table has NULL entries (gaps) between
  valid builtins. The original iteration broke on the first NULL,
  never reaching __builtin_riscv_attn at index 506. Fixed by
  skipping NULL/error_mark_node entries with a 5000-entry upper
  bound. (Note: this fix is historical — the current implementation
  uses inline asm instead of the builtin call.)

FIX 9: Control flow — code inserted after terminating branch
  gsi_last_bb(insert_bb) + gsi_insert_after placed the replacement
  code AFTER the block's terminating branch, making it unreachable.
  Fixed by using split_edge() to create a fresh basic block between
  the preheader and loop1's header. The new block has no terminator,
  so inserted code is properly reachable.

FIX 10: exit_bb NULL — single_exit() returns NULL
  single_exit(l4) returns NULL in GCC 15 for loops with complex
  exit structure or stale loop info. Added manual fallback that
  scans all blocks in loop 4 for edges leaving the loop. Second
  fallback: use the function's EXIT_BLOCK predecessor.

FIX 11: DCE removes builtin call (unused result)
  __builtin_riscv_attn() returns unsigned long, but the result
  was never used. Dead Code Elimination removed the call entirely.
  Fixed by switching to volatile inline assembly (.insn directive)
  which is immune to all optimization passes including DCE.

═══════════════════════════════════════════════════════════════
  FIXES APPLIED — CFG Replacement Correctness
═══════════════════════════════════════════════════════════════

FIX 10: ICE in calc_dfs_tree (stale dominance info)
  After redirect_edge_and_branch() the loop BBs were unreachable but still
  present in the CFG. update_ssa's DFS traversal crashed.
  Fix: call free_dominance_info(CDI_DOMINATORS) and
       free_dominance_info(CDI_POST_DOMINATORS) before returning.
  Forces update_ssa to recompute the dominator tree from scratch.

FIX 11: DCE crash in mark_operand_necessary (bb == NULL)
  SSA names defined in deleted loop BBs had NULL basic block pointers.
  DCE crashed dereferencing them. Virtual operand chain was also broken.
  Fix (three parts):
    a) Use flat arrays (ARRAY_REF + TREE_ADDRESSABLE) instead of COMPONENT_REF
       on dynamically-built struct types — avoids FRE's eliminate_stmt crash.
    b) Add "memory" clobber to the asm — creates VDEF anchoring the vop chain.
    c) Call mark_virtual_operands_for_renaming(cfun) — rethreads VUSEs to the
       new VDEF instead of deleted BBs.
    d) Return TODO_cleanup_cfg | TODO_update_ssa from execute() — cleanup
       removes dead BBs first; then update_ssa sees a consistent CFG.
       (Moving TODO_cleanup_cfg out of todo_flags_finish is essential —
        in finish flags it ran after SSA had already processed stale BBs.)

  SUMMARY
═══════════════════════════════════════════════════════════════

Two paths to the attn instruction:

  Path A: __builtin_riscv_attn(dims_addr, qkv_addr)
    → Direct, explicit. Works in all cases.
    → Output: attn a0,a0,a1

  Path B: Write plain C loops matching the attention pattern
    → Compiler auto-detects and replaces with attn instruction.
    → Requires: 4 separate stages (matmul, scale, softmax, matmul)
    → -O2 optimization level required for the pass to run.
    → Output: attn zero,a5,a4
